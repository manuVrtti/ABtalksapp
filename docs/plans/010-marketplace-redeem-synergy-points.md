# 010 ‚Äî Marketplace: redeem synergy points for ABTalks merchandise

## 1. Goal
Build the `/marketplace` page from the supplied mock ‚Äî a catalog of branded
items priced in synergy points (SP) that students can redeem. Five
coordinated pieces shipping together:

1. **`/marketplace` student route** ‚Äî hero + earning pills + sortable
   product grid + confirm-redeem flow.
2. **Atomic redemption** ‚Äî `redeemItemAction(itemId)` deducts SP and
   creates a `Redemption` row inside a transaction with a balance re-check
   (race-safe against rapid double-clicks and any concurrent SP-changing
   write).
3. **Admin `/admin/redemptions`** ‚Äî list of pending and historical
   redemptions with a "Mark fulfilled" action.
4. **Profile redemption history** ‚Äî a card on `/profile` showing the
   student's past redemptions and their status.
5. **Nav updates** ‚Äî add **Marketplace** as a 5th bottom-nav tab on mobile,
   and add the **"Refer and Redeem ‚Üí"** text link in the desktop header.

Catalog source: new `MarketplaceItem` table, seeded from
`prisma/content/marketplace.json`. Mirrors the existing pattern for
`problems.json` / `quizzes.json`.

SP values shown in the marketplace earning-pills row are pulled from
`scoring.ts` constants (so they stay in sync with `SYNERGY_BASE_SUBMISSION`,
`SYNERGY_REFERRAL`, `SYNERGY_PROOF_LINKEDIN`, `SYNERGY_PROOF_GITHUB` ‚Äî the
+10/+10/+7/+4 values in the mock are illustrative). The labels match the
mock: "Complete Task / Refer a Friend / Share on LinkedIn / Contribute on
GitHub".

**Step A1 ‚Äî Add schema**

Append to `prisma/schema.prisma`:

```prisma
enum RedemptionStatus {
  PENDING
  SHIPPED
  FULFILLED
  CANCELLED
}

model MarketplaceItem {
  id          String       @id @default(cuid())
  slug        String       @unique
  title       String
  description String
  costSP      Int
  imagePath   String?      // e.g. "/marketplace/classic-tee.png"
  active      Boolean      @default(true)
  sortOrder   Int          @default(0)
  createdAt   DateTime     @default(now())
  updatedAt   DateTime     @updatedAt
  redemptions Redemption[]

  @@index([active, sortOrder])
}

model Redemption {
  id               String           @id @default(cuid())
  userId           String
  itemId           String
  costSP           Int              // captured at redeem-time (history-safe)
  itemTitle        String           // captured at redeem-time (history-safe)
  status           RedemptionStatus @default(PENDING)
  shippingAddress  String           // free-text address block
  recipientPhone   String
  trackingNote     String?          // admin-set on SHIPPED transition
  createdAt        DateTime         @default(now())
  updatedAt        DateTime         @updatedAt
  user             User             @relation(fields: [userId], references: [id], onDelete: Cascade)
  item             MarketplaceItem  @relation(fields: [itemId], references: [id], onDelete: Restrict)

  @@index([userId, createdAt(sort: Desc)])
  @@index([status, createdAt(sort: Desc)])
}
```

Notes:
- **`costSP` and `itemTitle` are captured on the Redemption row** so
  history is stable even if the catalog item is later renamed or its
  price changes. **Do NOT** derive these from the live `MarketplaceItem`
  at read time.
- `onDelete: Restrict` on the item relation prevents accidental
  catalog-item deletion that would orphan redemption history. To remove
  an item, admin sets `active=false`.
- Add `redemptions Redemption[]` to the existing `User` model.

### Feature C ‚Äî Redemption flow

**Step C1 ‚Äî `redeem-item.ts`**

```ts
import { Prisma, RedemptionStatus } from "@prisma/client";
import { prisma } from "@/lib/db";

export type RedeemResult =
  | { ok: true; redemptionId: string; newBalance: number }
  | { ok: false; reason: "insufficient" | "inactive" | "not_found" | "no_profile" | "validation"; message: string };

export async function redeemItem(input: {
  userId: string;
  itemId: string;
  shippingAddress: string;
  recipientPhone: string;
}): Promise<RedeemResult> {
  try {
    return await prisma.$transaction(async (tx) => {
      const item = await tx.marketplaceItem.findUnique({
        where: { id: input.itemId },
        select: { id: true, title: true, costSP: true, active: true },
      });
      if (!item) return { ok: false, reason: "not_found", message: "Item not found" };
      if (!item.active) return { ok: false, reason: "inactive", message: "Item is no longer available" };

      const profile = await tx.studentProfile.findUnique({
        where: { userId: input.userId },
        select: { synergyPoints: true },
      });
      if (!profile) return { ok: false, reason: "no_profile", message: "Profile not found" };
      if (profile.synergyPoints < item.costSP) {
        return {
          ok: false,
          reason: "insufficient",
          message: `You need ${item.costSP - profile.synergyPoints} more SP for this item.`,
        };
      }

      const redemption = await tx.redemption.create({
        data: {
          userId: input.userId,
          itemId: item.id,
          costSP: item.costSP,
          itemTitle: item.title,
          status: RedemptionStatus.PENDING,
          shippingAddress: input.shippingAddress.trim(),
          recipientPhone: input.recipientPhone.trim(),
        },
        select: { id: true },
      });

      const updated = await tx.studentProfile.update({
        where: { userId: input.userId },
        data: { synergyPoints: { decrement: item.costSP } },
        select: { synergyPoints: true },
      });

      // Audit row in SynergyEvent
      await tx.synergyEvent.create({
        data: {
          userId: input.userId,
          points: -item.costSP,
          type: "REDEEM",
          reason: `Redeemed ${item.title} (redemptionId=${redemption.id})`,
        },
      });

      return { ok: true, redemptionId: redemption.id, newBalance: updated.synergyPoints };
    }, { maxWait: 5000, timeout: 10000 });
  } catch (error) {
    // Underflow shouldn't be possible because we checked, but guard anyway:
    if (
      error instanceof Prisma.PrismaClientKnownRequestError &&
      // Postgres CHECK violation if you later add CHECK (synergyPoints >= 0)
      false
    ) {
      return { ok: false, reason: "insufficient", message: "Balance changed; try again." };
    }
    throw error;
  }
}
```

Notes:
- **Balance check + decrement happen in the same transaction.** Even if
  the user double-clicks Redeem rapidly, the second tx reads the
  already-decremented balance.
- `costSP` and `itemTitle` are SNAPSHOTTED onto the Redemption row.
- A negative-points `SynergyEvent` of type `"REDEEM"` keeps the audit
  trail consistent with earn events. **Do NOT** skip this ‚Äî leaderboard
  recomputes / future reconciliations rely on `SynergyEvent` as the
  source of truth.

**Step C2 ‚Äî `marketplace-actions.ts`**

```ts
"use server";

import { auth } from "@/auth";
import { redeemItemSchema } from "@/lib/validations/marketplace";
import { redeemItem } from "@/features/marketplace/redeem-item";

export async function redeemItemAction(formData: FormData) {
  const session = await auth();
  if (!session?.user?.id) {
    return { ok: false as const, reason: "auth", message: "Sign in required" };
  }
  const parsed = redeemItemSchema.safeParse({
    itemId: formData.get("itemId"),
    shippingAddress: formData.get("shippingAddress"),
    recipientPhone: formData.get("recipientPhone"),
  });
  if (!parsed.success) {
    return {
      ok: false as const,
      reason: "validation" as const,
      message: parsed.error.issues[0]?.message ?? "Invalid input",
    };
  }
  return redeemItem({ userId: session.user.id, ...parsed.data });
}
```

**Step D1 ‚Äî `get-redemptions.ts`**

```ts
import { RedemptionStatus } from "@prisma/client";
import { prisma } from "@/lib/db";

export type AdminRedemptionRow = {
  id: string;
  createdAtIso: string;
  updatedAtIso: string;
  studentName: string;
  email: string;
  itemTitle: string;
  costSP: number;
  status: RedemptionStatus;
  shippingAddress: string;
  recipientPhone: string;
  trackingNote: string | null;
};

export async function getRedemptions(filter: {
  status?: RedemptionStatus | "ALL";
}): Promise<AdminRedemptionRow[]> {
  const rows = await prisma.redemption.findMany({
    where: filter.status && filter.status !== "ALL" ? { status: filter.status } : {},
    orderBy: [
      // PENDING first regardless of date ‚Äî admin's primary work queue
      { status: "asc" },
      { createdAt: "desc" },
    ],
    select: {
      id: true,
      createdAt: true,
      updatedAt: true,
      costSP: true,
      itemTitle: true,
      status: true,
      shippingAddress: true,
      recipientPhone: true,
      trackingNote: true,
      user: {
        select: {
          email: true,
          studentProfile: { select: { fullName: true } },
        },
      },
    },
    take: 500,
  });

  return rows.map((r) => ({
    id: r.id,
    createdAtIso: r.createdAt.toISOString(),
    updatedAtIso: r.updatedAt.toISOString(),
    studentName: r.user.studentProfile?.fullName?.trim() || r.user.email,
    email: r.user.email,
    itemTitle: r.itemTitle,
    costSP: r.costSP,
    status: r.status,
    shippingAddress: r.shippingAddress,
    recipientPhone: r.recipientPhone,
    trackingNote: r.trackingNote,
  }));
}
```

```ts
import { requireAdmin } from "@/lib/admin-auth";
import { prisma } from "@/lib/db";
import { updateRedemptionStatusSchema } from "@/lib/validations/marketplace";

export async function updateRedemptionStatusAction(formData: FormData) {
  await requireAdmin();
  const parsed = updateRedemptionStatusSchema.safeParse({
    redemptionId: formData.get("redemptionId"),
    nextStatus: formData.get("nextStatus"),
    trackingNote: formData.get("trackingNote") ?? undefined,
  });
  if (!parsed.success) {
    return { ok: false as const, message: parsed.error.issues[0]?.message ?? "Invalid input" };
  }
  const { redemptionId, nextStatus, trackingNote } = parsed.data;

  return prisma.$transaction(async (tx) => {
    const current = await tx.redemption.findUnique({
      where: { id: redemptionId },
      select: { status: true, userId: true, costSP: true },
    });
    if (!current) return { ok: false as const, message: "Redemption not found" };

    // Only allow forward transitions PENDING -> SHIPPED -> FULFILLED, or any -> CANCELLED.
    const allowed: Record<RedemptionStatus, RedemptionStatus[]> = {
      PENDING:   ["SHIPPED", "FULFILLED", "CANCELLED"],
      SHIPPED:   ["FULFILLED", "CANCELLED"],
      FULFILLED: [],
      CANCELLED: [],
    };
    if (!allowed[current.status].includes(nextStatus)) {
      return { ok: false as const, message: `Cannot transition ${current.status} ‚Üí ${nextStatus}` };
    }

    await tx.redemption.update({
      where: { id: redemptionId },
      data: { status: nextStatus, trackingNote: trackingNote ?? undefined },
    });

    // Refund SP on cancellation
    if (nextStatus === RedemptionStatus.CANCELLED && current.status !== RedemptionStatus.CANCELLED) {
      await tx.studentProfile.updateMany({
        where: { userId: current.userId },
        data: { synergyPoints: { increment: current.costSP } },
      });
      await tx.synergyEvent.create({
        data: {
          userId: current.userId,
          points: current.costSP,
          type: "REDEEM_REFUND",
          reason: `Refund for cancelled redemption ${redemptionId}`,
        },
      });
    }

    return { ok: true as const };
  });
}
```

## 6. Guardrails for Cursor (DO NOT)
- **DO NOT** allow the SP balance to go negative under any code path. The balance check MUST be inside the same transaction as the decrement (`Step C1`). A separate fetch + later decrement is a race condition.
- **DO NOT** derive `costSP` or `itemTitle` on the Redemption row from the live `MarketplaceItem` at read time. They are SNAPSHOTTED at create-time. Renaming an item or changing its price must NOT mutate historical redemptions.
- **DO NOT** skip the `SynergyEvent` audit row in either the redeem or the refund path. Every SP movement (positive or negative) gets a SynergyEvent. Future leaderboard/reconciliation logic depends on this invariant.
- **DO NOT** add a hard delete UI for `MarketplaceItem` (admin or otherwise). Use `active: false` to remove from the catalog. Hard deletes would orphan redemption history (`onDelete: Restrict` will throw anyway ‚Äî surface this as a guardrail so Cursor doesn't try).
- **DO NOT** set `recipientPhone` from the user's profile phone WITHOUT also showing it in the dialog editable. Some users redeem on a friend's account or to a different address; the dialog field defaults from profile but stays editable.
- **DO NOT** hardcode the +10/+10/+7/+4 SP values from the mock into `earning-pills.tsx`. The pills must interpolate from `scoring.ts` constants. Labels and icons match the mock; numbers come from code.
- **DO NOT** introduce email sending (toast confirmation only). The user explicitly scoped this out.
- **DO NOT** implement client-side sort in v1. The dropdown is visual; server sort by `sortOrder` is authoritative.
- **DO NOT** add inventory / stock tracking to `MarketplaceItem`. Out of scope.
- **DO NOT** add a redemption-per-user limit (e.g., "max 1 hoodie ever"). Out of scope.
- **DO NOT** wire the marketplace into the existing `AnalyticsDashboardLoader` or admin overview as a chart. Add the link as a stat card on the overview if natural, but no analytics charts in this plan.
- **DO NOT** alter `synergy-chip.tsx`, `scoring.ts`, or any synergy backend logic in this plan. The header chip refreshes via `router.refresh()` after a successful redeem.
- **DO NOT** rely on Vercel Blob, Cloudinary, or any image upload service. v1 uses static `public/marketplace/*.png` assets. Image upload tooling is a separate plan.
- **DO NOT** swap the existing `Sparkles` synergy chip icon to `Flame` in this plan ‚Äî that's plan 005's territory. If plan 005 has already landed, the chip is already Flame and you don't need to do anything.
- **DO NOT** modify `bottom-nav.tsx`'s route-exclusion regex (`/^\/(login|register|claude-signup|students)(\/|$)/`). The marketplace IS protected and SHOULD show the bottom nav.
- **DO NOT** delete or merge `prisma/migrations/` history. Each migration stands as the historical record. Generate a NEW migration; do not edit prior migration SQL.
- **DO NOT** skip the `costSP >= 0` check at the validation layer (Zod). Even though the catalog ships with positive values, a future schema-level addition (e.g., a free item with cost 0) should still work, but a NEGATIVE cost must be rejected.
